## Section 4 — Recommendation

### Fields to add to `VideoAnalyticsResult` (HIGH value, reliably returned)

1. **`engagementScore`** (0–100 scale) — from `/statistics?videoGuid=`. Already returned for our test video as `50`. This is the single most actionable metric Bunny provides: a composite viewer retention score. **Add this.**

2. **`countryViewCounts`** — from `/statistics?videoGuid=`. Real data: `{"IN": 12}`. Country-level breakdown of views by ISO country code. Our current `topCountries` field in `VideoAnalyticsResult` already models this. **Already surfaced — verify it matches.**

3. **`countryWatchTime`** — from `/statistics?videoGuid=`. Real data: `{"IN": 4446}`. Watch time broken down by country. **New field — add this alongside countryViewCounts.**

4. **`viewsChart`** / **`watchTimeChart`** — from `/statistics?videoGuid=`. Daily time-series of views and watch time. These are ideal for trend charts on the admin dashboard. **Add as optional time-series data for detail views.**

5. **`averageWatchTime`** — from Video Object. Real data: `370` seconds. This is distinct from `avgWatchPercent` (which we compute). Raw seconds is useful. **Add this.**

### Fields that look tier-gated — skip unless Mihir upgrades Bunny plan

1. **Video Play Data** (`/videos/{id}/play`) — returned **401 Authentication denied**. This likely requires a higher-tier plan or a different auth mechanism (Account-level API key vs Library-level). **Skip for launch.**

2. **Video Heatmap Data** (`/videos/{id}/heatmap-data`) — returned **404**. This is a newer endpoint documented in Bunny API but may not be available on the free trial. The legacy `/heatmap` endpoint works fine. **Skip — use legacy heatmap.**

3. **Video Storage Size Info** (`/videos/{id}/storage-size`) — returned **404**. May not exist yet or may require a paid plan. **Skip.**

### Fields that exist but have no real value on our data

1. **`category`** — always `"unknown"` for our uploads. Bunny auto-categorization doesn't work on our content type.
2. **`chapters`**, **`moments`**, **`captions`** — all empty arrays. We don't upload chapters or captions.
3. **`framerate`**, **`width×height`**, **`availableResolutions`** — technically correct (30fps, 3840×2160, 360p-1080p) but not meaningful for admin analytics dashboards.
4. **`storageSize`** — 8.6 GB for one video. Interesting for cost monitoring but not admin analytics.
5. **`engagementScore` at library level** — returns `-1` (meaning "not enough data across all videos"). Only useful per-video.

### Key discovery: Heatmap format difference

Our current `IVideoProvider.getVideoHeatmap()` expects `heatmap: number[]` (array). But the real Bunny API returns `heatmap: { "0": 100, "1": 72, ... }` (object with string keys mapping to percentage values, where keys are video positions in seconds). 

**Action required:** Verify that `bunny-video.provider.ts` correctly transforms this object format into the `number[]` that the interface expects. If it doesn't, this is a silent data corruption bug.

### Summary: Recommended next step

Add **3 new fields** to `VideoAnalyticsResult`:
- `engagementScore: number` (0–100, or -1 if insufficient data)
- `countryWatchTime: Record<string, number>` (seconds by country)
- `averageWatchTime: number` (seconds, from video object)

And optionally expose **2 time-series fields** for detail views:
- `viewsChart: Record<string, number>` (date → count)
- `watchTimeChart: Record<string, number>` (date → seconds)

**Do NOT add:** play data (auth-gated), heatmap-data (404), storage-size (404), category/chapters/moments (empty).

**Priority check before any code changes:** Verify the heatmap format transformation in `bunny-video.provider.ts`. If it's passing the raw object through as-is, the frontend is receiving `{0: 100, 1: 72}` instead of `[100, 72, ...]`.
